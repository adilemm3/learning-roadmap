# HTTP Clients & IHttpClientFactory — Реальные кейсы

## Кейс 1: Port exhaustion в продакшне

**Ситуация:** Сервис обработки заказов работает нормально при низкой нагрузке. При пиковой нагрузке (Black Friday) начинают сыпаться `SocketException: Only one usage of each socket address is permitted`.

**Причина:** `new HttpClient()` создавался на каждый запрос в `foreach` по заказам. При 500 req/sec за 4 минуты накапливается 120 000 сокетов в TIME_WAIT — больше чем лимит ОС.

**Решение:** Заменить на `IHttpClientFactory` с typed client. Один handler на все запросы к одному хосту.

**Урок:** Port exhaustion не проявляется при низкой нагрузке — только в продакшне на пике.

## Кейс 2: Retry storm при падении внешнего сервиса

**Ситуация:** Платёжный шлюз лёг на 5 минут. Все микросервисы начали делать retry → нагрузка на шлюз выросла в 4 раза → шлюз не мог восстановиться.

**Причина:** Retry без Circuit Breaker. Каждый из 10 инстансов сервиса делал 3 retry → 40 запросов вместо 10 при каждом вызове.

**Решение:** Добавить Circuit Breaker снаружи Retry. После 5 провалов — Open state на 30 сек. Шлюз восстановился за 2 минуты вместо 15.

**Урок:** Retry без CB — вежливый DoS на упавший сервис.

## Кейс 3: Дублирование платежей при retry

**Ситуация:** При таймауте запроса к платёжному шлюзу retry отправлял платёж повторно. Клиенты получали двойные списания.

**Причина:** Retry настроен на все ошибки включая таймаут. Но при таймауте первый запрос мог уже выполниться на стороне шлюза.

**Решение 1:** Retry только на сетевые ошибки (connection refused, DNS), не на таймаут и 5xx.
**Решение 2:** Добавить `Idempotency-Key` заголовок — шлюз дедуплицирует повторные запросы с одним ключом.

**Урок:** Идемпотентность — обязательное требование для retry в финансовых операциях.

## Кейс 4: DNS-проблема со static HttpClient в Kubernetes

**Ситуация:** После rolling update сервиса некоторые поды не могли достучаться до обновлённого сервиса — `Connection refused`. Рестарт старых подов решал проблему.

**Причина:** `static HttpClient` кэшировал IP старого пода. Kubernetes изменил IP при rolling update, но handler не ротировался.

**Решение:** Заменить `static HttpClient` на `IHttpClientFactory`. Ротация каждые 2 минуты подхватывает новые IP из Kubernetes DNS.

**Урок:** В Kubernetes IP подов меняются при каждом деплое — static HttpClient несовместим с K8s.
